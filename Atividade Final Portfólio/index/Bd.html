Banco de Dados Relacional

Essa foi uma das primeiras atividades que fizemos onde relacionamos duas tabelas através de uma entidade de relacionamento, atribuimos as Primary Keys que servem para registrar individualmente cada item em uma tabela, atribuimos as cardinalidades ao diagrama entidade relacionamento, atribuimos também as Foreign Key que é possível identificar pelos simbolos de "#" antes do nome.




Criamos esse diagrama com base em uma sequencia de requisitos que precisavamos cumprir, por exemplo, se o cliente quiser sber a quantidade de alunos de tal curso, qual curso é, disciplinas optativas e obrigatórias, entre outros pontos. Vale destacar que os requisitos são um ponto impórtante na elaboração de um diagrama pois atraves deles poderemos criar as relações e as consultas no banco de dados. Assim como a imagem anterior, aplicamos os mesmos conceitos porém acrescido da foreign key que estão presentes em uma entidade de relacionamento e na tabela de Alunos, a importancia da Foreign key surge quando não se é possível estabeçer uma relação entre duas tabelas se a tabela não possuir a Foreign Key, em uma relação entre tabelas onde a cardinalidade é N:N existe a necessidade de criarmos uma tabela a mais, onde contem as duas Foreign Keys para podermos estabelecer a relação entre as tabelas.




Assim como o exercicio anterior, foi necessário a criação de uma tabela a mais que seria a nossa entidade de relacionamento, isso se dá pois em um banco de dados relacional, quando temos uma relação muitos-para-muitos (N N) entre duas entidades, é necessário criar uma tabela intermediária, chamada de entidade de relacionamento ou tabela de junção (junction table), para gerenciar essa relação. Existem as Normalizações de tabelas, chamadas de Formas normais, na teoria de normalização, não é permitido ter uma relação muitos para muitos entre duas tabelas, a separação da relação em duas relações 1:N permite uma melhor consulta no banco de dados, também tem a questão de integridade referencial, quando separamos em duas tabelas garantimos que as relações entre as entidades sejam consistentes.




Aqui podemos ver os comandos DML (Data manipulation language), essa classe de comandos se caracteriza pelos comandos Insert, Update, Delete, Select. Também podemos ver os comandos Create, que são comandos DDL (Data definition language), os comandos DDL são Create, Alter e Drop. Para utilizar os comandos Create inserimos o comando seguido do nome da tabela, por exemplo: Create table cachorro, assim criando a tabela cachorro. Cada comando existe uma sintaxe de uso que podemos encontrar facilmente em uma rápida pesquisa.




Nessa imagem, vemos as consultas realizadas em um banco de dados relacional, os comandos Select, selecionam a coluna que gostariamos, de acordo com o From que seria a tabela, para selecionar um cachorro da tabela animais seria o seguinte comando: Select cachorro From Animais, os comandos de Select e From fazem parte dos comandos DQL (Data query Language), porém, existem outros do tipo DQL como Inner join.




Nessa imagem, criamos o diagrama mais dificil da matéria, tivemos várias dificuldades porém deu tudo certo! Criamos um banco de dados para um sistema educacional, onde temos os alunos, disciplinas optativas e obrigatorias, cursos disponiveis e departamentos. Para realizar essa atividade depois de elaborar o diagrama, criamos o Query utilizamos os comandos, DML, DQL e DDL, criamos as tabelas, inserimos os dados nas tabelas, criamos as consultas e a maior dificuldade foi criar as consultas onde usamos os diversos comandos ensinados durante o semestre, segue um exemplo de comando utilizado para apresentar a quantidade de alunos ativos por curso:

SELECT nomeCurso, COUNT(*) AS quantidade FROM (matriculas NATURAL INNER JOIN estudantes) INNER JOIN cursos USING (idCurso) WHERE graduado=false GROUP BY nomeCurso;
